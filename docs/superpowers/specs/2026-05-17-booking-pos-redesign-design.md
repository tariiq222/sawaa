
# إعادة تصميم صفحة الحجز — نمط POS

التاريخ: 2026-05-17
المشروع: apps/dashboard — مركز سواء للاستشارات الأسرية (single-tenant)

## الهدف

استبدال نظام الـ wizard متعدد الخطوات لإنشاء حجز جديد بشاشة واحدة بنمط POS: كل الحقول مرئية دفعة واحدة في عمودين — فورم كثيف + ملخص حجز ثابت (sticky) — لتسريع عمل موظفة الاستقبال.

## السياق الحالي

- المودال القديم تحوّل سابقاً إلى view كامل داخل صفحة الحجوزات (`booking-create-view.tsx`).
- الـ wizard الحالي (`booking-wizard.tsx`) فيه 4 خطوات منطقية و`useWizardState` يدير `step` + `chosenPath` + منطق التنقل (goBack/jumpToStep).
- إعداد `flowOrder` من backend (`service_first` / `employee_first` / `both`) يتحكم في ترتيب الخطوات.
- مكونات الخطوات: `booking-client-step.tsx`, `wizard-steps/step-service.tsx`, `step-employee.tsx`, `step-type-duration.tsx`, `step-datetime.tsx`, `step-confirm.tsx`, `step-choose-path.tsx`, `step-scheduling.tsx`.
- `createMut` من `hooks/use-bookings.ts` ينفّذ الإنشاء.

## التصميم

### التخطيط — POS بعمودين

شاشة واحدة داخل صفحة الحجوزات (نفس مكان الـ view الحالي):
- **عمود الفورم (الأكبر، ~2/3):** كل الأقسام الخمسة مرئية دفعة واحدة بكثافة عالية، بطاقات صغيرة، نقرة واحدة لكل اختيار، scroll لأدنى حد. لا أزرار "التالي/السابق".
- **عمود الملخص (الأصغر، ~1/3، sticky):** ملخص الحجز الحي — المستفيد، الخدمة، الممارس، النوع، الموعد، تفصيل السعر (الخدمة + كوبون − خصم = الإجمالي)، خيار الدفع بالعيادة، وزر "تأكيد الحجز" كبير أسفله.

### أقسام عمود الفورم (كلها مرئية، كثيفة)

1. المستفيد — بحث/اختيار عميل + إضافة جديد.
2. الخدمة — شبكة بطاقات مضغوطة.
3. الممارس — شبكة بطاقات مضغوطة.
4. النوع والمدّة — أزرار النوع (حضوري/عن بعد/زيارة) + اختيار المدّة.
5. الموعد — تقويم + سلوتات الأوقات المتاحة.

### المرونة (موظفة استقبال، ليست عميل)

لا قفل ترتيبي صارم. كل قسم متاح، الاستقبال تعبّي بأي ترتيب يناسبها. الترتيب أعلاه افتراضي بصري فقط.

### التبعيات

- الخدمة والممارس مستقلان — أيهما أولاً.
- المدّة تعتمد على الخدمة — بدون خدمة يعرض القسم تلميح "اختر خدمة" بدل أزرار فاضية.
- السلوتات تعتمد على (ممارس + خدمة + نوع) — ناقص شيء يعرض قسم الموعد تلميح خفيف "اختر الخدمة والممارس لعرض الأوقات"؛ تكتمل التبعيات → السلوتات تُجلب تلقائياً.
- العميل مستقل تماماً.

### التبسيط

يُلغى `flowOrder` و`chosenPath` و`StepChoosePath` نهائياً. الترتيب البصري ثابت (مستفيد→خدمة→ممارس→نوع→موعد) لكنه غير إجباري.

### الحالة

`useWizardState` → `useBookingFormState` (`use-booking-form-state.ts`): نفس حقول `WizardState` ناقص `step` و`chosenPath`؛ setters عادية بدون منطق تنقّل (لا goBack/jumpToStep/goToStep).

### التحقق

زر "تأكيد الحجز" يكون disabled حتى تكتمل الحقول الإلزامية: مستفيد، خدمة، ممارس، نوع، تاريخ، وقت. الحقل الناقص في الملخص يظهر "—" بلون باهت.

### معالجة الأخطاء

- فشل جلب السلوتات → رسالة خطأ صغيرة داخل قسم الموعد + إعادة محاولة (موجود في `useBookingSlots`).
- فشل `createMut` → `toast.error` كما هو حالياً، الفورم يبقى معبّى.
- نجاح → reset + `onSuccess` (رجوع لقائمة الحجوزات).

## الملفات

**جديد:**
- `components/features/bookings/booking-pos.tsx` — الحاوية بعمودين.
- `components/features/bookings/booking-summary.tsx` — عمود الملخص الـ sticky.
- `components/features/bookings/use-booking-form-state.ts` — حالة الفورم.

**يُعدّل:**
- `booking-create-view.tsx` — يستدعي `BookingPos` بدل `BookingWizard`.
- مكونات اختيار العميل/الخدمة/الممارس/النوع/الموعد — تُحوّل لنُسخ مضغوطة (بطاقات أصغر، grid أكثر أعمدة) لتناسب كثافة POS.

**يُحذف:**
- `booking-wizard.tsx`, `use-wizard-state.ts`, `wizard-steps/step-choose-path.tsx`, و`wizard-steps/step-confirm.tsx` و`step-scheduling.tsx` إن لم تُعد للاستخدام.

## ملاحظة صريحة

مكونات الخطوات الحالية مصممة كخطوة كاملة العرض. تحويلها لكثافة POS يتطلب تعديلها فعلياً (بطاقات أصغر، أعمدة أكثر) — هذا أوسع من "تغيير الحاوية فقط".

## القيود

- RTL منطقي فقط (`ps-/pe-/ms-/me-/border-e`)، لا left/right.
- ألوان من التوكنز فقط، لا hex.
- لا inline styles، أيقونات `@hugeicons` فقط.
- حدود حجم الملفات: feature component ≤300، hook ≤200، أي ملف ≤350.
- كل نص عبر `t()` من `useLocale()`، AR/EN parity مطلوبة.
- `createMut` يبقى كما هو.
