## Which handler creates the rows

Even though no handler was modified during the spike, the live source code in `set-employee-durations.handler.ts` shows the duplication-creating pattern. The handler iterates `cmd.durations[*].items[]` and for each item WITHOUT an `id` it calls `tx.serviceDurationOption.create(...)` unconditionally — there is **no pre-check for an existing `(serviceId, deliveryType, employeeServiceId, durationMins)` row**.

**File:** `apps/backend/src/modules/org-experience/services/set-employee-durations/set-employee-durations.handler.ts`
**Line range:** 83–101 (the create branch)

```ts
// Create new row
const created = await tx.serviceDurationOption.create({
  data: {
    serviceId: cmd.serviceId,
    deliveryType,
    employeeServiceId,
    label: item.label,
    labelAr: item.labelAr,
    durationMins: item.durationMins,
    price: item.price,
    currency: 'SAR',
    isDefault: false,
    sortOrder: 0,
    isActive: true,
  },
});
```

The lookup-by-id path (lines 57–82) only matches when the client supplied `item.id`. If the dashboard sends a "fresh" copy of the existing row (no `id`), the handler silently creates a new row instead of updating. There is no upsert and no `findFirst({ serviceId, deliveryType, employeeServiceId, durationMins })` guard before the create.

A structurally identical gap exists in:

- `set-duration-options.handler.ts` line 53 — `tx.serviceDurationOption.create` for items with no `id`
- `set-employee-custom-pricing/set-employee-custom-pricing.handler.ts` line 57 — `tx.serviceDurationOption.create` when no `anchor` is found
- `set-service-booking-configs.handler.ts` line 121 — `tx.serviceDurationOption.create` after `deleteMany`

None of these were modified in the spike window, but they were all live and running on production on 2026-06-23.

**Schema also lacks a defensive constraint.** `\d "ServiceDurationOption"` shows the only unique constraint is `WHERE "isDefault" = true AND "isActive" = true` (one default per delivery type). There is no `UNIQUE (serviceId, deliveryType, durationMins)` or partial unique on active rows, so the database itself does not prevent the duplication.

### What this tells us

- The duplication pattern is **NOT in the seed** (8/8 unique on local).
- The duplication pattern is **NOT visible from local DB evidence alone** (Query 2 = 0 rows).
- The duplication pattern must therefore be triggered by **production-only code paths**: the admin "edit practitioner durations" UI calling `SetEmployeeDurationsHandler` repeatedly (or in a loop, or with stale state), without sending the existing `item.id` back to the server.
- Combined with the no-handler-change git history, the 2026-06-23 spike is consistent with **repeated invocations of an existing handler** — not a new code push, not a migration, not a one-off script.

## Verdict

**`recurring-handler-call` (high confidence).**

**Reasoning:**

1. **No code change in the spike window.** None of the ~95 commits in 2026-06-22 → 2026-06-24 touched `ServiceDurationOption` handlers, the Prisma model, the schema, or seeds. Two migrations were applied but both target unrelated tables (`Client` trigram index, `CustomRole.permissionsCustomized`). This rules out `one-off-deploy-script`.

2. **No schema-level prevention.** The `ServiceDurationOption` table lacks a unique constraint on `(serviceId, deliveryType, durationMins)` (only on the `isDefault` partial). The DB will accept duplicates indefinitely.

3. **Handler-side gap is live.** `SetEmployeeDurationsHandler.execute()` creates a new row whenever the client posts a duration option without an `item.id`. If the dashboard re-fetches the options after a previous save but loses the IDs (race condition, stale cache, refetch-into-form) and then re-submits, each save creates N new rows instead of updating N. A 61-row spike in a single day is consistent with an admin session editing the same set of durations 4–6 times.

4. **Local seed does not reproduce.** Production has 102 duplicates across 18 triples; local has 0. The duplication requires production-only triggers — primarily the admin UI flow that posts to `PATCH /api/v1/dashboard/services/set-employee-durations` (or equivalent) repeatedly.

5. **`manual-DB-write` is unlikely.** A 61-row one-day insert with consistent `(serviceId, deliveryType, durationMins)` shape and varying prices/`labelAr` looks like form submissions, not a `psql` paste or admin tool batch.

The probable trigger sequence (specific date/laptop still unknown — flag for follow-up):

- Manager opens "edit practitioner durations" for a service+employee.
- Saves durations → 1 set of rows created (with prices).
- Reopens the page → form may be re-loaded from a stale cache or from the wrong scope (e.g., service-defaults view instead of practitioner-owned view), so the existing IDs are missing.
- Saves again → a new identical set of rows is created on top of the old ones.
- Repeats 4–6 times → spike.

---

## Implication for fix design

The spike is **architectural, not a one-time cleanup.** Two follow-up tasks should be addressed in the plan:

1. **DB-level guard (P1):** add a partial unique constraint on `(serviceId, deliveryType, employeeServiceId, durationMins)` filtered by `isActive = true`. This stops the bleeding immediately and matches the existing pattern used for `isDefault` uniqueness. Note: this requires a deduplication migration first to clear current 102 dup rows; the plan should sequence a "merge duplicates, keep cheapest/latest" step BEFORE the constraint is added. Migration is immutable — must be a new migration.

2. **Handler-level guard (P1):** in `SetEmployeeDurationsHandler`, `SetDurationOptionsHandler`, and `SetEmployeeCustomPricingHandler`, before `tx.serviceDurationOption.create`, do a `findFirst({ serviceId, deliveryType, employeeServiceId, durationMins })` and either return the existing row (idempotent upsert) or throw a `409 Conflict` so the dashboard can refresh its form. This makes the API safe-by-default even if the constraint migration is delayed.

A third, smaller item: the dashboard form should preserve `item.id` across re-fetches (e.g., when the user toggles "use practitioner pricing" on/off, the owned rows should round-trip). This is a frontend fix but blocks the same root cause. Defer to a frontend ticket once the backend is hardened.
