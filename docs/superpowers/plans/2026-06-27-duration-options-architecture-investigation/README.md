# Service Pricing Architecture Investigation — Index

**Status:** Investigation complete (date: 2026-06-27)
**Goal:** Evidence base for designing the "ideal" fix for ServiceDurationOption duplication.

## Findings

| # | Topic | TL;DR |
|---|---|---|
| 1 | [2026-06-23 Spike](findings/01-spike-2026-06-23.md) | No commit in the 2026-06-22 → 2026-06-24 window touched any ServiceDurationOption handler, model, schema, or seed, yet live `SetEmployeeDurationsHandler.execute()` (line 83–101) creates a new row whenever the client posts an item without an `id` and there is no DB-level UNIQUE on `(serviceId, deliveryType, durationMins)`; verdict is `recurring-handler-call` with high confidence — a duplicated save 4–6 times in one admin session produces the observed spike. |
| 2 | [Schema Archaeology](findings/02-schema-archaeology.md) | All three pricing tables (ServiceBookingConfig, ServiceDurationOption, EmployeeServiceOption) are live and load-bearing; `ServiceBookingConfig` is the delivery-channel gate + INHERIT-mode step 3 fallback, `ServiceDurationOption` is the customer-facing duration/price menu (and gained `employeeServiceId` ownership on 2026-06-19), `EmployeeServiceOption` is the per-practitioner override (INHERIT-mode step 1), and `EmployeeService.useCustomPricing` (added 2026-06-20) is the boolean that switches the resolver between CUSTOM and INHERIT row sets — and the only UNIQUE on `ServiceDurationOption` is partial on `isDefault=true AND isActive=true`, which explains why no schema guard has ever prevented the duplication. |
| 3 | [Frontend Call Map](findings/03-frontend-call-map.md) | The dashboard uses endpoint #2 (`PUT /dashboard/organization/services/:id/booking-types`) and endpoint #4 (`PUT /dashboard/people/employees/:id/services/:serviceId/durations`) for all pricing writes; endpoints #1 (`set-duration-options`) and #3 (`set-employee-custom-pricing`) are mounted in their controllers but have no frontend caller — orphaned — and the service-edit PUT embeds duration options inline into the booking-types payload, so `SetServiceBookingConfigsHandler` is the only handler the main service-edit save ever hits for `ServiceDurationOption` writes. |
| 4 | [Booking Flow Trace](findings/04-booking-flow-trace.md) | The customer-facing flow charges exactly what the choice step displays: `PractitionerChoicePicker` reads from `GET /public/services/:serviceId/practitioners/:employeeId/booking-options`, which composes the same priority chain (`EmployeeServiceOption.priceOverride` → `ServiceDurationOption.price` → `ServiceBookingConfig.price` → `Service.price`) that `PriceResolverService.resolve()` re-applies server-side on `POST /public/bookings` — both gated on `EmployeeService.useCustomPricing` to switch between CUSTOM (owned rows only) and INHERIT (service-defaults + override) row sets, and the only "drift" is a benign two-screen price walk between the pre-therapist service picker and the post-override choice step. |
| 5 | [Service Edit Flow Trace](findings/05-service-edit-flow-trace.md) | The verified duplication trigger is `SetServiceBookingConfigsHandler` at line 121 (`prisma.serviceDurationOption.create` runs unconditionally when `option.id` is absent, the form's `buildDurationOptionsPayload` does not dedupe by `(durationMins, price, deliveryType)`, and the upstream `deleteMany` at line 95 only filters by `id NOT IN optionIds` — not by triple equality — so a payload that includes a non-default option matching the synthetic default's `(deliveryType, durationMins)` produces two rows for the same triple); the secondary trigger is `SetEmployeeDurationsHandler` (C9, practitioner-owned scope, line 83–101 + line 105–114) with the same pattern; `UpdateServiceHandler`, the image-upload path, and `SetEmployeeServiceOptionsHandler` do not touch `ServiceDurationOption` and so cannot create duplicates there. |

## Open Questions

- **Production audit log / Sentry check for the exact 2026-06-23 trigger.** Task 1 produced a high-confidence verdict of `recurring-handler-call` from local-only evidence (no commit in the spike window touched the handler, and the structural duplicate pattern is reproducible from the handler code alone); the specific date/laptop/session that produced the 61 rows still requires pulling the production audit log or Sentry events for 2026-06-23 to confirm which admin session saved which service+employee how many times. Flagged for follow-up.
- **`Booking.durationOptionId` cascade behavior on `ServiceDurationOption` deletion.** Task 5 §4c + §8 noted that `SetServiceBookingConfigsHandler`'s `deleteMany` at line 95 can delete rows that existing `Booking` rows reference via `Booking.durationOptionId`; the migration history in Task 2 was not re-parsed to confirm whether the FK is nullable or whether the relation is `onDelete: SetNull` / `onDelete: Restrict` — affects whether the dedupe migration in Tier 2 is safe to run as a raw DELETE without first nulling booking references.
- **`mergeDraftsFromServer` / `get-service-booking-configs` employeeServiceId filter.** Task 5 §6 confirmed that the dashboard GET returns practitioner-owned rows mixed into the per-config `durationOptions` array (no `employeeServiceId` filter at handler line 178), and that the form's `mergeDraftsFromServer` (`booking-types-editor.tsx:226`) only filters out `isDefault`, not `employeeServiceId IS NULL` — so a user on the service-level pricing tab can see and edit practitioner-owned rows, and the PUT then mutates them via `updateMany` at handler line 104 (which has no `employeeServiceId` filter either). The right fix is to add `employeeServiceId: null` to BOTH the read filter and the handler's updateMany/deleteMany; not yet decided which side to fix first.
- **`SetEmployeeDurationsHandler` "soft-deactivate leftovers" keep-vs-leftover math.** Task 5 §5b found that the keep-vs-leftover math at line 105–114 uses `id` equality, not `(serviceId, deliveryType, durationMins, employeeServiceId)` equality — so if two rows with the same 4-tuple both made it into `keepIds` (which is what happens on the create branch at line 83–101 when no `id` is supplied and the same triple is added twice), neither is deactivated. Lower blast radius than the service-level trigger but the same root cause.
- **`PriceResolverService` inactive-row fallback behavior.** Task 4 §8 noted that the practitioner-facing options endpoint and the resolver both filter `isActive=true`, so a `SetEmployeeDurationsHandler` "soft-deactivate" (line 105–114, `isActive=false`) cannot be re-surfaced by the customer flow — but the resolver side of this claim (what it does when an inactive row is the only match for an INHERIT-mode fallback) was not re-read in Task 5.
- **Mobile booking tree divergence from the website wizard.** Task 4 §7 spot-checked `apps/mobile/app/(client)/booking` and confirmed the mobile tree does not implement the choice step (it posts `serviceId/employeeId/slot/deliveryType` without a practitioner-options fetch), so the server-side resolution still applies correctly but mobile customers never see an `EmployeeServiceOption` override reflected in the choice UI. Not a bug — server-side billing is correct — but a UX inconsistency that may warrant a follow-up.
- **`SetEmployeeCustomPricingHandler` (orphaned, #3) and the `20260520150000_finalize_delivery_type_transition` partial-index skip behavior.** Task 1 confirmed the handler is orphaned (no UI caller) and Task 2 confirmed the migration historically skipped creating `ServiceDurationOption_one_default_active_delivery_idx` when duplicate active defaults exist; whether the partial index is now present in production (i.e., whether prod's current state would re-trigger the DO-skip) was not queried directly. Affects Tier 1's "add partial UNIQUE on `(serviceId, deliveryType, durationMins) WHERE isActive=true`" migration — needs a pre-check on the prod DB.
- **`EmployeeServiceOption` consolidation vs coexistence with the newer `ServiceDurationOption.employeeServiceId` ownership pattern.** Task 2 §3 (EmployeeServiceOption verdict) noted that both patterns achieve the same outcome (practitioner-specific menu) and may be candidates for consolidation; the architectural decision (keep both, fold one into the other, or unify on a single per-practitioner override table) is open for Task 7's synthesis.
